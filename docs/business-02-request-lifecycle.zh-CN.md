# Kode 请求全链路：一条需求是怎样被执行出来的

这篇文档只回答一个问题：当用户在 Kode 里输入一句话后，系统内部到底发生了什么。

## 先看一张总图

```mermaid
flowchart TB
    U[用户输入]
    A[入口层<br/>REPL / print / ACP]
    B[上下文装配<br/>README Git AGENTS 目录 会话]
    C[系统提示词增强]
    D[模型请求]
    E{模型是否发起工具调用}
    F[直接回复用户]
    G[工具队列]
    H[权限检查]
    I[执行工具]
    J[把工具结果写回对话]
    K[继续推理]
    L[会话落盘与状态更新]

    U --> A --> B --> C --> D --> E
    E -- 否 --> F --> L
    E -- 是 --> G --> H --> I --> J --> K --> D
```

## 第 0 步：用户从哪个入口进来

```mermaid
flowchart LR
    A[kode] --> A1[交互式 REPL]
    B[kode -p] --> B1[非交互式 ask]
    C[kode-acp] --> C1[通过 JSON-RPC 对外提供 Agent 能力]
```

这三种入口最后都会汇聚到同一套编排逻辑，只是交互方式不同。

## 第 1 步：系统先决定“这句话是什么类型”

用户输入到达后，不会立刻交给模型，而是先分流。

```mermaid
flowchart TD
    A[收到输入]
    B{是否为 slash command}
    C[本地命令或命令面板]
    D{是否包含 @mention}
    E[注入提醒上下文]
    F[普通自然语言请求]
    G[进入统一 AI 查询链路]

    A --> B
    B -- 是 --> C
    B -- 否 --> D
    D -- 是 --> E --> G
    D -- 否 --> F --> G
```

### 这里的三种输入含义

- `/xxx`：走命令系统，比如 `/model`、`/plugin`、`/mcp`
- `@ask-*`：触发专家模型咨询倾向
- `@run-agent-*`：触发子代理委派倾向
- `@file/path`：提示系统用户正在引用特定文件

## 第 2 步：系统拼装“模型真正看到的上下文”

这一步是项目最关键的业务价值之一。模型拿到的不是裸问题，而是“问题 + 项目现场”。

```mermaid
mindmap
  root((进入模型前的上下文))
    用户当前问题
    既有消息历史
    README
    Git 状态
    目录结构快照
    AGENTS.md / AGENTS.override.md
    CLAUDE.md 兼容内容
    项目配置 context
    系统提醒
```

### 系统为什么要主动补上下文

因为用户经常不会把背景讲完整，真正的业务价值来自系统主动知道：

- 当前项目是什么
- 现在 Git 工作区是什么状态
- 团队要求代理遵守什么规则
- 之前已经聊到哪一步

## 第 3 步：系统提示词不是固定文案，而是动态增强

```mermaid
flowchart LR
    A[基础 System Prompt]
    B[项目上下文]
    C[提醒信息]
    D[模型特定增强]
    E[最终送入模型的 Prompt]

    A --> E
    B --> E
    C --> E
    D --> E
```

动态增强的价值在于：

- 不同模型可以收到不同的补充提示
- 提及文件/agent/model 时能得到额外提醒
- 项目规范能稳定注入，而不是依赖用户重复说明

## 第 4 步：模型开始推理，并决定是否调用工具

从业务上讲，模型在这里扮演的是“调度中心”，不是“执行者本体”。

```mermaid
sequenceDiagram
    participant U as 用户
    participant S as Kode 编排层
    participant M as 模型
    participant T as 工具层

    U->>S: 输入需求
    S->>M: 发送增强后的上下文与问题
    M-->>S: 回复文本 或 tool_use 指令
    alt 只需回答
        S-->>U: 输出答案
    else 需要执行
        S->>T: 调用具体工具
        T-->>S: 返回结果
        S->>M: 把工具结果回填给模型
        M-->>S: 继续推理或生成最终答案
        S-->>U: 输出最终结果
    end
```

## 第 5 步：工具不是随便调用，而是进入工具队列

项目里存在一个非常明确的“工具队列”概念，这意味着系统把工具调用当作可调度任务来管理，而不是临时函数调用。

```mermaid
flowchart TB
    A[模型发起工具调用]
    B[解析工具名与参数]
    C[判断是否并发安全]
    D[进入队列]
    E{是否可立即执行}
    F[等待中]
    G[执行工具]
    H[写回结果]

    A --> B --> C --> D --> E
    E -- 否 --> F
    E -- 是 --> G --> H
```

### 这里的业务意义

- 可以避免多个危险工具同时乱跑
- 只让“可并发的安全工具”并发执行
- 执行中可以给用户显示进度

## 第 6 步：每个工具都会经过权限与输入校验

```mermaid
flowchart TD
    A[工具请求]
    B[输入 Schema 校验]
    C[工具级权限判断]
    D{当前模式}
    E[YOLO/宽松模式]
    F[Safe 模式]
    G[Plan 模式]
    H[自动放行 或 告警]
    I[请求用户批准]
    J[只允许读和计划相关动作]

    A --> B --> C --> D
    D --> E --> H
    D --> F --> I
    D --> G --> J
```

### 项目里的工具家族

```mermaid
flowchart LR
    A[工具层]
    A --> B[文件工具]
    A --> C[系统工具]
    A --> D[搜索工具]
    A --> E[网络工具]
    A --> F[AI 工具]
    A --> G[MCP 工具]
    A --> H[交互工具]

    B --> B1[Read Edit Write Notebook]
    C --> C1[Bash KillShell TaskOutput]
    D --> D1[Grep Glob Lsp]
    E --> E1[WebFetch WebSearch]
    F --> F1[AskExpert Skill Task]
    G --> G1[ListMcpResources ReadMcpResource MCPTool]
    H --> H1[AskUserQuestion SlashCommand TodoWrite]
```

## 第 7 步：工具执行结果不会直接结束，而是会回填给模型继续思考

很多人会误以为“工具执行完就结束了”，其实不是。工具结果只是中间材料。

```mermaid
flowchart LR
    A[工具结果]
    B[转换成模型可理解的消息]
    C[附加回对话历史]
    D[模型继续推理]
    E[决定下一步]

    A --> B --> C --> D --> E
```

下一步可能是：

- 继续调用别的工具
- 输出用户可读答案
- 让子代理接手
- 进入等待或中断状态

## 第 8 步：如果需要，还会调用子代理或专家模型

这是 Kode 从“单体助手”走向“协作系统”的关键点。

```mermaid
flowchart TB
    A[主对话]
    B{是否需要更专门的处理}
    C[AskExpertModel]
    D[TaskTool 子代理]
    E[独立处理]
    F[结果返回主线程]

    A --> B
    B -- 咨询特定模型 --> C --> E --> F
    B -- 委派子任务 --> D --> E --> F
```

### 两种协作方式的区别

| 方式 | 本质 | 适合什么 |
|---|---|---|
| `@ask-*` / AskExpertModel | 临时咨询另一个模型 | 获得第二视角、专家意见 |
| `@run-agent-*` / TaskTool | 拉起具备角色设定与工具边界的子代理 | 分工执行、复杂子任务 |

## 第 9 步：会话会被持续保存，不是“一次性回答”

系统会把消息落到 JSONL 日志中，并维护 session id、slug、tag、title 等元数据。

```mermaid
flowchart LR
    A[用户消息]
    B[助手消息]
    C[工具结果]
    D[JSONL 会话文件]
    E[支持 resume / continue / fork]

    A --> D
    B --> D
    C --> D
    D --> E
```

这意味着业务上它不是“瞬时响应产品”，而是“持续任务产品”。

## 第 10 步：上下文过长时，系统会自动压缩记忆

```mermaid
stateDiagram-v2
    [*] --> 正常对话
    正常对话 --> 检查 Token 阈值
    检查 Token 阈值 --> 正常对话: 未超过阈值
    检查 Token 阈值 --> 自动压缩: 超过阈值
    自动压缩 --> 生成摘要
    生成摘要 --> 恢复关键文件内容
    恢复关键文件内容 --> 继续对话
    继续对话 --> [*]
```

### 自动压缩的业务意义

- 让长会话不至于因为上下文过长而崩掉
- 尽量保留关键状态而不是硬清空
- 让“长期任务”有持续性

## 一条完整请求的生命周期

```mermaid
journey
    title 一条请求的业务生命周期
    section 输入
      用户描述需求: 4: 用户
      系统识别命令或普通请求: 5: Kode
    section 理解
      读取项目上下文: 5: Kode
      增强 system prompt: 5: Kode
      模型制定处理策略: 4: 模型
    section 执行
      选择工具: 5: 模型
      进行权限判断: 4: Kode
      执行工具: 5: Kode
    section 收敛
      工具结果回填: 5: Kode
      模型生成最终回答: 5: 模型
      结果显示给用户: 5: Kode
    section 沉淀
      写入会话日志: 5: Kode
      支持后续恢复继续: 5: Kode
```

## 这一整条链路里，最值得你从业务上盯住的点

### 1. 上下文装配质量

它决定系统是不是“真的懂项目”，而不只是“回答得像懂”。

### 2. 工具选择质量

它决定模型能不能把抽象需求真正落地成动作。

### 3. 权限与队列治理

它决定自动化能力会不会变成风险。

### 4. 会话稳定性

它决定产品能不能承接长任务、复杂任务、多人协作任务。

### 5. 子代理与专家协作

它决定系统有没有从单 Agent 升级为多 Agent 编排平台的潜力。

## 最后用一句话总结

Kode 的请求处理链路，本质上是：

“把自然语言需求，经过上下文增强、模型决策、工具执行、权限治理和会话沉淀，转化成一个可持续的终端工作流。”
